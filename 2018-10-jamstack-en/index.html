Gatsby

Gatsby is a static site generator, built with React, that takes content sources, transforms it into a static GraphQL endpoint, which you can use to fetch data into your templates.

What is GraphQL?

In short, GraphQL is a query language with which you can fetch data from a single endpoint, specifying your data query and receiving a JSON response in the same shape of the query.

What is a headless CMS?

A CMS where the presentation (head) is decoupled from the data layer (body).

Differences to traditional CMSes

  • Does not "render" any content or pages itself
  • Exposes the content via an API (REST, GraphQL, JSON)
  • Tends to have more focus on the authoring experience
  • Often is cloud-based (CMS-as-a-service, Content-as-a-service)
  • Stores content in a "pure" format, decoupled from a web context

Why choose headless?

  • No sole focus on web content
  • Teams can start instantly, work together simultaneously
  • Front-end devs can pick perfect exactly what's needed
  • Fits right into a microservices architecture
  • Can be used as one of many sources of content
  • Flexibility, freedom, ease of setup
  • Not limited by web page content (social media, mobile apps, flight information, entertainment, sports, etc.)
  • FE, BE, author teams can work cooperatively to incrementally work on new features.
  • They can use preferred technologies instead of having to learn CMS internals.
  • Everything is decoupled and can be linked with an API.
  • There's no restriction to web-like content or data. Can be any content, any CMS, etc.

Why not choose headless?

  • Lack of focus on web-like features
  • Some projects may contain a lot of "moving parts"
  • Content editing is not WYSIWYG
  • Marketers may need more developer assistance
  • SEO, page architecture, asset linking. Makes some features more challenging
  • for certain features, like security, link management, asset management, search, e-commerce, etc.
  • as content is primitive and has no immediate preview front-end
  • for creating page layouts and customer journeys on a website

When choose a headless CMS?

  • Smaller short-lived projects low-stake marketing websites
  • Large projects with multi-channel content
  • Organizations with microservices-oriented architecture
  • low-stake marketing websites
  • where the content is not web-only (social media, apps, websites, information systems)
  • that need a CMS service

What does JAMStack solve in these situations?

  • No overhead running a CMS / framework
  • Reduced complexity of hosting and server
  • No laggyness caused by heavy APIs/CMSes
  • Improved security of static files
  • More freedom with decoupled front-ends
  • Files worden gegenereerd bij deployment
  • Elke hosting die een bak bestanden kan serveren is voldoende
  • Bezoeker heeft geen last van externe services omdat deze aangesproken worden bij deployment
  • Omdat er geen CMS of framework bij een request aan te pas komt, zijn er ook veel minder/geen aanvalshoeken
